# Media Catalog V1

Status: approved for implementation. Direction in V1: Android provides a catalog; Windows queries and displays it. The catalog transfers metadata and bounded thumbnails only. Original file bytes are sent later through File Transfer V1/V2.

## Capability and security

The base capability is `media-catalog-v1`. A peer MUST send catalog query, page, change, permission, error, cancel, or download messages only after an authenticated encrypted connection has negotiated this capability. Thumbnail request/response additionally require `thumbnails-v1`, so a provider can expose metadata without supporting previews. `thumbnailAvailable` is treated as false unless both capabilities were negotiated. Defining a capability in code is not sufficient: an application MUST advertise it only after the corresponding handler is usable.

The Android provider exposes only items and collections explicitly allowed by the user. It MUST NOT expose Android filesystem paths, `content://` URIs, EXIF GPS coordinates, or provider-internal identifiers. `itemId`, `albumId`, page tokens, and revisions are opaque protocol values.

## Messages

| Type | Direction | Purpose |
|---|---|---|
| `catalog.query` | Windows → Android | Request one filtered and sorted page |
| `catalog.page` | Android → Windows | Return a page from one catalog snapshot |
| `catalog.changed` | Android → Windows | Notify about incremental changes or require refresh |
| `catalog.thumbnail.request` | Windows → Android | Request a bounded preview |
| `catalog.thumbnail.response` | Android → Windows | Return a Base64-encoded preview |
| `catalog.file.download.request` | Windows → Android | Ask Android to offer the original through File Transfer |
| `catalog.permission` | Android → Windows | Report granted, limited, denied, or revoked access |
| `catalog.error` | Android → Windows | Reject an operation with a stable error code |
| `catalog.cancel` | either direction | Idempotently cancel a request |

All messages use the normal DeviceSync envelope. `queryId` identifies a logical paged query. `requestId` identifies one thumbnail, download, or cancellable operation. IDs MUST be UUID strings generated by the requester.

## Metadata

`CatalogItemPayload` has these camelCase fields:

| Field | Type | Rules |
|---|---|---|
| `itemId` | string | Stable opaque ID while the provider item exists |
| `displayName` | string | User-visible name, never a path; 1–255 Unicode scalars |
| `mimeType` | string | Lowercase MIME type or `application/octet-stream` |
| `sizeBytes` | int64/null | Null when a cloud provider cannot report it |
| `createdAtUtc` | ISO-8601/null | UTC timestamp when known |
| `modifiedAtUtc` | ISO-8601 | UTC timestamp |
| `category` | string | `image`, `video`, `audio`, `document`, or `other` |
| `width`, `height` | int32/null | Positive pixel dimensions when applicable |
| `durationMillis` | int64/null | Non-negative duration for audio/video |
| `albumId` | string/null | Opaque collection identifier |
| `albumDisplayName` | string/null | Display-only album/folder name, never a path |
| `generation` | int64 | Monotonically increasing provider generation |
| `revision` | string | Opaque version token for stale-item detection |
| `thumbnailAvailable` | boolean | Whether a preview can currently be generated |

Unknown fields MUST be ignored. Missing required fields MUST reject the payload. Receivers MUST validate all semantic limits after deserialization.

## Query, sorting, and pagination

`catalog.query` contains `queryId`, `pageSize`, optional opaque `pageToken`, `categories`, `mimeTypes`, optional `albumId`, optional `search`, `sortBy`, `sortDirection`, optional `modifiedAfterUtc`, and optional `generationAfter`.

- `pageSize` MUST be 1–200; default is 100.
- Allowed `sortBy`: `modifiedAtUtc`, `createdAtUtc`, `displayName`, `sizeBytes`.
- Allowed `sortDirection`: `asc`, `desc`.
- Category and MIME filters are combined with AND; values within each list use OR.
- A page token is scoped to its original filters, ordering, permission set, and snapshot. It MUST NOT contain a readable path or secret.
- The requester sends the next query only after receiving the previous `catalog.page` or `catalog.error`. This one-page-in-flight rule provides backpressure.

`catalog.page` contains `queryId`, `items`, nullable `nextPageToken`, `snapshotGeneration`, and `hasMore`. `hasMore=true` requires a non-empty token. Pages in one traversal MUST retain a stable ordering. An expired or incompatible token produces `INVALID_PAGE_TOKEN`; Windows restarts from the first page.

## Incremental changes

`catalog.changed` contains `generation`, `changes`, and `requiresRefresh`. Each change contains `itemId`, `changeType` (`added`, `updated`, or `removed`), and optional current `revision`.

Android coalesces repeated changes and sends at most 200 entries per notification. If changes are lost, permissions change, or the limit is exceeded, it sends `requiresRefresh=true`; the list may then be empty. Windows discards affected cached pages and queries again. A generation MUST never move backwards within one provider identity.

Rename is `updated` with a new revision. Removal is `removed`; future thumbnail or download requests return `ITEM_NOT_FOUND`. Windows MUST key cached items by `itemId`, not display name.

## Thumbnails

The request contains `requestId`, `itemId`, optional `expectedRevision`, `maxWidth`, `maxHeight`, `format`, and `quality`.

- Width and height MUST each be 32–512 pixels.
- Formats in V1 are `jpeg` and `webp`; quality MUST be 40–90.
- The decoded thumbnail MUST be at most 256 KiB and the JSON payload must also fit the negotiated frame limit.
- Android MUST preserve aspect ratio and MUST NOT upscale the source.
- Data is standard padded Base64 in JSON for V1.
- Windows may have at most four thumbnail requests outstanding. Android may return `BUSY` with `retryable=true` instead of queueing unbounded work.
- A stale `expectedRevision` returns `ITEM_CHANGED` and the current generation. No stale preview is returned.

Cancellation is best-effort and idempotent. A response racing with a cancellation may arrive; the requester discards it. Android checks cancellation before decode, while loading, and before serialization.

## Original download

`catalog.file.download.request` contains `requestId`, `itemId`, optional `expectedRevision`, and a requester-generated `transferId`. If the item and revision still exist, Android opens the provider stream only then and sends `file.offer` with exactly that `transferId`. Existing file-transfer acceptance, streaming, hash verification, cancellation, size limits, and one-active-transfer rules apply. Catalog download MUST NOT bypass the Windows acceptance policy.

Cloud-only items may fail with `ITEM_UNAVAILABLE` or may require provider download; Android MUST report a retryable error rather than buffering the original in memory.

## Permission

`catalog.permission` contains optional `requestId`, `state`, `grantedCategories`, `canRequest`, and optional `reasonCode`. States are `granted`, `limited`, `denied`, `revoked`, and `unavailable`.

On revoke Android MUST cancel catalog work, stop sending metadata, invalidate page tokens, clear thumbnail cache entries derived from revoked access, and send `catalog.permission` with `state=revoked`. Windows clears corresponding cached metadata and previews. The protocol cannot open Android permission UI remotely; `canRequest=true` only tells Windows that it may instruct the user to act on the phone.

## Errors and cancellation

`catalog.error` contains optional `requestId`, stable `code`, optional user-safe `message`, `retryable`, and optional `currentGeneration`. Codes in V1:

- `UNSUPPORTED`, `PERMISSION_REQUIRED`, `PERMISSION_REVOKED`
- `INVALID_QUERY`, `INVALID_PAGE_TOKEN`, `INVALID_REQUEST`
- `ITEM_NOT_FOUND`, `ITEM_CHANGED`, `ITEM_UNAVAILABLE`
- `THUMBNAIL_UNAVAILABLE`, `THUMBNAIL_TOO_LARGE`
- `BUSY`, `RATE_LIMITED`, `CANCELLED`, `INTERNAL_ERROR`

`catalog.cancel` contains `requestId` and `reason`. Repeating it has no side effects. Cancellation of an unknown or completed request is treated as success and does not produce a protocol error.

## Test-vector invariants

Both platforms MUST deserialize the canonical vectors under `protocol/test-vectors/media-catalog/`, preserve 64-bit numeric values, use the exact camelCase names and message type strings, ignore `futureField`, and reject missing required fields. No vector or runtime payload may contain a filesystem path or `content://` URI.
